1. Object-Oriented Programming (OOP)

Object-Oriented Programming is a software design paradigm that models real-world concepts as code structures called objects. An object contains state data (attributes) and behaviors (methods).

Classes vs. Objects

  • Class: A template or blueprint that defines the structure and potential behavior of an object.
  • Object: A concrete instance created (instantiated) from a class blueprint, carrying its own independent state.

The Four Pillars of OOP

Mastering OOP requires understanding four foundational architectural concepts:

  1. Encapsulation: Bundling data and methods into a single unit (class) and restricting direct external access to internals using private visibility modifiers (getters/setters).
  2. Abstraction: Hiding internal implementation complexity and exposing only a clean, simplified interface (using interfaces or abstract classes).
  3. Inheritance: Creating a parent-child relationship between classes, allowing a child class to inherit and extend attributes/methods from a parent.
  4. Polymorphism: The ability for different objects to respond to the same method call in unique ways (method overriding or interface implementation).
Python OOP Classes & Inheritance example 
class Animal:
    def __init__(self, name):
        self._name = name # Protected attribute (Encapsulation)
        
    def speak(self):
        pass             # Abstract method outline

# Dog inherits from Animal (Inheritance)
class Dog(Animal):
    def speak(self):
        return f"{self._name} says Woof!" # Polymorphism

my_dog = Dog("Buddy")
print(my_dog.speak()) # Buddy says Woof!

2. Data Structures

A data structure is a specialized format for organizing, storing, and retrieving data. Different data structures are optimized for different computational operations.

Linear Data Structures

  • Array: A collection of elements stored sequentially in contiguous memory. Offers instant random access (via indexes) but holds a fixed size.
  • Linked List: A chain of elements (nodes) where each node points to the next. Easily resized dynamically, but requires sequential scanning to find values.
  • Stack: A Last-In, First-Out (LIFO) structure. Think of a stack of plates—you add (push) and remove (pop) only from the top.
  • Queue: A First-In, First-Out (FIFO) structure. Like a queue at a grocery store—the first to enter (enqueue) is the first to leave (dequeue).

Non-Linear Data Structures

  • Hash Table / Dictionary: Stores key-value pairs. Uses a hashing function to map keys directly to memory bins, providing ultra-fast O(1) reads and writes.
  • Binary Search Tree (BST): A hierarchical tree structure where each parent node has at most two children. The left child is smaller, and the right child is larger, enabling efficient search operations.
  • Graph: A network of nodes (vertices) connected by links (edges). Used to model social networks, city pathways, or web links.
💡 Trade-Offs are Key

No data structure is perfect for everything. An Array gives you immediate reads, but deleting or inserting in the middle is slow. A Linked List allows instant insertions, but finding a specific value takes time. Choosing the right data structure is a core software engineering skill.

3. Algorithms & Computational Complexity

An algorithm is a step-by-step procedure or formula for solving a computational problem. How we measure if an algorithm is "good" or "bad" is evaluated using Big O notation.

Understanding Big O Notation

Big O describes how the execution time (Time Complexity) or memory usage (Space Complexity) of an algorithm scales as the input size (denoted as N) grows towards infinity.

Big O Notation Common Name Description Algorithm Example
O(1) Constant Time Operation takes the same time, regardless of input size. Accessing array element by index; Hash Table lookup.
O(log N) Logarithmic Time Input is halved on each step. Highly efficient. Binary Search in a sorted list.
O(N) Linear Time Time scales directly proportion to the size of the input. Linear Search across an unsorted list.
O(N log N) Linearithmic Time Dividing step (log N) performed N times. Standard for sorting. Merge Sort, Quick Sort.
O(N²) Quadratic Time Nested iterations. Scales poorly with larger inputs. Bubble Sort, Selection Sort.

Binary Search vs. Linear Search

Consider finding a number in a sorted array of 1,000,000 elements:

  • Linear Search (O(N)): Scans element-by-element. In the worst case, it performs 1,000,000 operations.
  • Binary Search (O(log N)): Looks at the middle element, discarding half of the list on each step. It takes a maximum of only 20 operations!
Python Binary Search O(log N) search implementation 
def binary_search(arr, target):
    low = 0
    high = len(arr) - 1
    
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid # Found! Return index
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
            
    return -1 # Not found

4. Memory Management (Stack vs. Heap)

When programs run, they request memory allocation from the operating system. This temporary memory is partitioned into two distinct physical zones: The Stack and The Heap.

The Stack

  • How it works: Managed automatically using a LIFO structure. It stores local function variables, parameters, and return addresses.
  • Lifespan: Strictly bounded to the execution of the function. Once a function returns, its "stack frame" is automatically discarded.
  • Pros/Cons: Extremely fast allocation, but small and rigid in size.

The Heap

  • How it works: A large, unstructured pool of memory used for variables that must survive beyond local function calls or require a dynamic size (e.g., dynamic arrays, objects).
  • Lifespan: Programmatic. Deallocation must be requested manually or handled by an automated runtime process.
  • Pros/Cons: Large and highly flexible, but slower to access and allocate.

Manual Allocation vs. Garbage Collection

How heap memory is reclaimed is language dependent:

  • Manual Memory Management (C/C++): The developer explicitly allocates (e.g., malloc() or new) and must manually free memory (e.g., free() or delete). Forgetting to free memory leads to Memory Leaks, while freeing it too early causes Dangling Pointers.
  • Garbage Collection (Java, JS, Python): A background engine automatically tracks objects and reclaims their memory when there are no remaining references pointing to them. This saves developers time but can introduce periodic performance pauses (GC sweeps).
C++ Manual Allocation Explicit heap management 
// Allocating an array on the Heap manually
int* numbers = new int[500]; 

// ... perform computations ...

// MUST free the memory when finished to avoid a memory leak
delete[] numbers;

5. Robust Error & Exception Handling

Real-world systems inevitably encounter errors: networks drop, files go missing, and users enter invalid data. A robust program handles these anomalies gracefully instead of crashing abruptly.

Traditional Error Codes vs. Exceptions

  • Error Codes: Classic languages (like C) return status codes from functions (e.g., -1 for failure). Developers must explicitly check this code after every call, leading to cluttered logic.
  • Exceptions: Modern languages separate error handling from the main business logic by "throwing" (or raising) an exception object, which propagates up the call stack until it is intercepted by a "catch" handler.

The Try-Catch-Finally Mechanism

Most modern languages utilize a unified try-catch block structure:

  1. Try: Wraps a block of suspect code that might produce an exception.
  2. Catch: Intercepts and handles specific exceptions thrown by the try block.
  3. Finally: A block that runs unconditionally at the end, regardless of whether an exception occurred or was caught. Crucial for cleaning up active file handlers or network links.
Java Exceptions Graceful failure flow 
try {
    int result = 10 / 0; // Throws ArithmeticException
} catch (ArithmeticException e) {
    System.out.println("Error: Division by zero!");
} finally {
    System.out.println("Execution finished, resources cleaned.");
}
🏆 Exceptional Engineering Tip

Always catch the most specific exceptions first before falling back on general/generic catch blocks. This allows you to handle unique errors (like file-not-found) differently from others (like memory limits), ensuring the application behaves predictably.
